## English

### Before You Start

- Check existing [Issues](https://github.com/staklab/PrismRail/issues) before opening a new one.
- For large features, UI changes, or behavior changes, open an issue first.
- Keep changes aligned with PrismRail's local-first and low-intrusion design principles.

### How To Contribute

1. Fork the repository or create a branch from `main`.
2. Make focused changes with clear scope.
3. Run the project checks before opening a pull request:

```bash
npm run check
```

4. Open a pull request with a clear summary of what changed and why.

### Pull Request Guidelines

- Keep permissions minimal.
- Do not introduce remote code execution or unnecessary network behavior.
- Preserve privacy-first behavior whenever possible.
- Update documentation when features, supported sites, or setup steps change.
- Prefer small, reviewable pull requests over large mixed changes.

### Issues

Use Issues for:

- bug reports
- supported-site breakages
- UI or UX suggestions
- feature requests
- documentation problems

Open a new issue here: [New Issue](https://github.com/staklab/PrismRail/issues/new/choose)

## 中文

### 开始之前

- 提交前请先查看已有 [Issues](https://github.com/staklab/PrismRail/issues)。
- 如果是较大的功能、UI 调整或行为变更，建议先开 Issue 讨论。
- 改动请尽量保持 PrismRail 本地优先、低侵入的设计原则。

### 如何贡献

1. Fork 仓库，或从 `main` 创建分支。
2. 进行范围清晰、目标单一的修改。
3. 提交 PR 前先运行检查：

```bash
npm run check
```

4. 发起 Pull Request，并清楚说明改了什么、为什么改。

### Pull Request 规范

- 尽量保持最小权限原则。
- 不要引入远程代码执行或不必要的网络行为。
- 尽量保持隐私优先、本地优先的使用方式。
- 功能、适配站点或安装步骤有变化时，请同步更新文档。
- 尽量把 PR 控制在易于 review 的范围内。

### Issues 使用场景

以下情况建议直接提 Issue：

- Bug 反馈
- 站点适配失效
- UI / UX 优化建议
- 功能需求
- 文档问题

新建 Issue 入口：[New Issue](https://github.com/staklab/PrismRail/issues/new/choose)
