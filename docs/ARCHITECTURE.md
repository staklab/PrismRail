# 技术架构

## 1. 设计原则

- 安全优先：尽量避免破坏宿主站点内部渲染状态
- 模块清晰：站点识别、数据抽取、性能优化、UI 渲染彼此解耦
- 可扩展：新增站点时只新增适配器，不改核心逻辑
- 低依赖：首版零第三方依赖，方便开源与跨浏览器分发

## 2. 总体结构

- `background/`：后台初始化与默认配置
- `options/`：设置页，用于调节阈值与功能开关
- `adapters/`：站点适配器
- `core/`：共享基础能力、会话调度、存储封装
- `performance/`：渲染优化策略
- `timeline/`：右侧时间轴 UI
- `ui/`：插件注入样式

## 3. 运行流程

1. 内容脚本进入目标站点
2. 注册表选择对应适配器
3. 轮询加观察器识别消息节点
4. 提取消息文本、角色、索引与摘要
5. 应用性能优化策略
6. 渲染时间轴并绑定跳转
7. 监听滚动与 DOM 变更，持续增量刷新

## 4. 模块边界

### 4.1 Adapter Layer

- 输入：当前 `location.hostname` 与页面 DOM
- 输出：标准化消息记录数组
- 职责：屏蔽不同站点 DOM 差异

### 4.2 Core Session

- 输入：适配器输出与用户设置
- 输出：统一的插件生命周期管理
- 职责：调度扫描、刷新、观察器和事件监听

### 4.3 Performance Layer

- 输入：消息节点数组
- 输出：被增强的节点与样式状态
- 职责：离屏优化、代码块折叠、媒体惰性提示

### 4.4 Timeline Layer

- 输入：标准化消息记录
- 输出：右侧可交互时间轴
- 职责：固定长度时间轴、12 节点采样、导航、预览、激活态同步

## 5. 性能优化方案

## 5.1 MVP 采用

- `content-visibility: auto`
- `contain: layout style paint`
- `contain-intrinsic-size`
- 超长代码块折叠
- 图片 `loading="lazy"` 与 `decoding="async"` 提示
- DOM 刷新节流与 `requestAnimationFrame` 调度

## 5.2 暂不默认启用

- 直接移除历史消息 DOM
- 消息 HTML 快照归档
- 侵入式重排宿主页面结构

原因：这些方案虽然理论上更强，但在 React/复杂富文本环境中更容易触发兼容性问题。

## 6. 时间轴数据结构

```js
{
  id: "chatgpt-12",
  index: 12,
  role: "assistant",
  label: "A13",
  excerpt: "这里展示本地截取的前 80 到 120 个字符。",
  node: HTMLElement
}
```

时间轴节点采样规则：

- 固定渲染 12 个点。
- 第 1 个点：首对话起点。
- 第 2 到第 11 个点：分别映射整体内容进度的 10%、20% 直到 100%，并跳转到对应分段起点。
- 第 12 个点：最后一段回复的尾部，用于快速落到整段对话的结束位置。

## 7. 跨站点适配策略

- 每个站点维护一组优先选择器
- 失败时回退到通用消息节点启发式
- 角色识别优先读取宿主站点显式属性
- 无法识别时采用文本与索引回退策略
- 当前内置：ChatGPT、Gemini、Claude、DeepSeek、豆包、GLM、Grok

## 8. 移动端与 iPad 适配

- 时间轴宽度压缩
- 预览气泡最大宽度随视口缩小
- 减少视觉噪音，保留关键点击目标
- 默认右移内边距更大，避免误触系统滚动区域

## 9. 跨浏览器策略

- 以 Chromium Manifest V3 为主线
- Safari 通过 Web Extension 转换流程接入
- 所有核心逻辑使用原生 DOM API，不依赖浏览器私有功能
- 存储层通过 `chrome` / `browser` 兼容包装
- 响应式布局覆盖窄屏和 iPad 宽度，但移动端是否允许安装扩展仍取决于浏览器平台能力

## 10. 测试建议

- 单元测试：摘要生成、适配器注册表
- 手工回归：ChatGPT/Gemini/Claude 长会话
- 设备回归：桌面端、窄屏、iPad 宽度
- 性能观察：节点数量上升后滚动流畅度与布局稳定性
